# Triage（分类）

通过一组小的 triage 角色状态机流转项目 issue tracker 上的 issues。

如果这个 repo 把外部 pull requests 当作请求面（见 issue-tracker 配置），triage 也覆盖它们：**一个 PR 是一个带附加代码的 issue**——相同的角色、相同的状态、相同的机器，带几个标为 "for a PR" 的增量。根据 tracker 配置把裸的 `#42` 解析为 issue 或 PR。

在 triage 期间发布到 issue tracker 的每个评论或 issue **必须**以这个免责声明开头：

```
> *This was generated by AI during triage.*
```

## 角色

两个**分类**角色：

- `bug` —— 某物坏了
- `enhancement` —— 新功能或改进

五个**状态**角色：

- `needs-triage` —— 维护者需要评估
- `needs-info` —— 等待报告者更多信息
- `ready-for-agent` —— 完全规格化，准备好给 AFK agent
- `ready-for-human` —— 需要人工实现
- `wontfix` —— 不会被处理

对于 PR，相同的状态读作针对附加代码：`ready-for-agent` 意味着附加了简报，agent 应该对 diff 采取下一步；`ready-for-human` 意味着它准备好让人合并。

每个 triaged 的 issue 应该恰好携带一个分类角色和一个状态角色。如果状态角色冲突，标记它并在做任何事之前问维护者。

这些是规范角色名称——issue tracker 中使用的实际标签字符串可能不同。映射应该已经提供给你了——如果没有，运行 `/setup-matt-pocock-skills`。

状态转换：未标记的 issue 通常先到 `needs-triage`；从那里它移动到 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。`needs-info` 在报告者回复后返回 `needs-triage`。维护者可以随时覆盖——标记看起来不寻常的转换并在继续之前问。

## 调用

维护者调用 `/triage` 并用自然语言描述他们想要什么。解释请求并行动。例子：

- "Show me anything that needs my attention"
- "Let's look at #42"（issue 或 PR）
- "Move #42 to ready-for-agent"
- "What's ready for agents to pick up?"

## 显示需要注意的

查询 issue tracker 并按 oldest first 呈现三个桶：

1. **未标记** —— 从未 triage 过。
2. **`needs-triage`** —— 评估进行中。
3. **`needs-info` 带报告者自上次 triage 笔记以来的活动** —— 需要重新评估。

当 PR 在范围内时，在这些桶中包含外部 PR 并标记每行 `[PR]` 或 `[issue]`。发现面只包括*外部* PR（tracker 配置定义谁是外部的）—— 协作者的进行中 PR 不是 triage 工作。这个过滤器只是发现面的；明确命名的 PR 总是被 triage 无论作者。

显示计数和每个项目的一行总结。让维护者选。

## Triage 特定 issue 或 PR

1. **收集上下文。** 读完整 issue 或 PR（正文、评论、标签、作者、日期；对于 PR，还有 diff）。解析任何之前的 triage 笔记这样你不会重新问已解决的问题。使用项目的领域词汇探索代码库，尊重该区域的 ADR。对代码库运行两个检查：(a) **冗余** —— 按领域概念（不只是请求的措辞）搜索请求行为的已有实现，并报告你在哪找的。如果找到了，它是一个已实现的 `wontfix`（步骤 5）。(b) **之前拒绝** —— 读 `.out-of-scope/*.md` 并暴露任何类似这个请求的。

2. **推荐。** 告诉维护者你的分类和状态推荐及推理，加上与请求相关的代码库简短总结——包括它是否已实现。等指示。

3. **验证声明。** 在任何追问之前，检查声明是否成立。对于 bug，从报告者的步骤复现它。对于 PR，确认 diff 做了它声称的——检出它，运行相关测试或命令。报告发生了什么：确认（带代码路径）、失败、或细节不足（一个强的 `needs-info` 信号）。一个确认的验证让 agent 简报更有说服力。

4. **追问（如果需要）。** 如果请求需要充实，一起运行 `/grilling` 和 `/domain-modeling` skills——一次一个问题把它追问成形，精化领域术语并在决策落地时内联更新 `CONTEXT.md`/ADRs。

5. **应用结果：**
   - `ready-for-agent` —— 发布 agent 简报评论（[AGENT-BRIEF.md](AGENT-BRIEF.md)）。
   - `ready-for-human` —— 与 agent 简报相同结构，但注明为什么不能委托（判断调用、外部访问、设计决策、手动测试）。
   - `needs-info` —— 发布 triage 笔记（模板如下）。
   - `wontfix` —— 关闭，评论取决于*为什么*：
     - **已实现** —— 变更已存在于代码库中。指向它住在哪；**不要**写到 `.out-of-scope/`（那个 KB 是给*被拒绝的*请求的，不是已构建的）。
     - **拒绝（bug）** —— 礼貌解释，然后关闭。
     - **拒绝（enhancement）** —— 写到 `.out-of-scope/`，从评论链接到它，然后关闭（[OUT-OF-SCOPE.md](OUT-OF-SCOPE.md)）。
   - `needs-triage` —— 应用角色。如果有部分进展可选评论。

## 快速状态覆盖

如果维护者说 "move #42 to ready-for-agent"，信任他们并直接应用角色。确认你即将做什么（角色变更、评论、关闭），然后行动。跳过追问。如果移动到 `ready-for-agent` 而没有追问会话，问他们是否想写 agent 简报。

## Needs-info 模板

```markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
```

在 "established so far" 下捕获追问期间解决的所有东西这样工作不会丢。问题必须具体且可操作，不是 "please provide more info"。

## 恢复之前的会话

如果 issue 或 PR 上存在之前的 triage 笔记，读它们，检查报告者是否回答了任何未决问题，并在继续之前呈现更新的画面。不要重新问已解决的问题。
